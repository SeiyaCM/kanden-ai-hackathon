# スタックちゃん独り言に Home Assistant の疲労度を反映する

## What We're Building

**目的:** AI_StackChan2 の **独り言モード**（定期的に `exec_chatGPT` へ送る短い話題）において、**Home Assistant 上の疲労度**（例: `sensor.kanden_fatigue`）を参照し、**スコア帯に応じた話題・トーン**で会話を始める。

**既存との関係:**

- リポジトリでは疲労度は **MQTT `kanden/fatigue` → `sensor.kanden_fatigue`**（`homeassistant/mqtt_kanden_fatigue.yaml`）。
- スタックちゃん連携は **HA → HTTP（`curl`）** が既採用（`package_stackchan_fatigue.yaml`）。いまの独り言は **デバイス単体**の `random_words[]` タイマー（`AI_StackChan2` `main.cpp`）。
- 本件は **「デバイスが HA から値を読む」** 方向が自然で、既存の **HA がスタックちゃんへプッシュ**するパターンとは **データ流れが逆**になる。

**成功イメージ（確定したトーン）:**

- **低疲労:** 明るい冗談や軽い雑談をしやすい話題に寄せる。
- **高疲労:** 寄り添い・共感を中心とした話題に寄せる（からかいや軽薄な冗談は控えめにする）。

## Why This Approach（検討の軸）

- **単一情報源:** 疲労の「正」はすでに HA に集約されているため、スタックちゃん側で二重計測しない。
- **YAGNI:** まずは **取得経路を 1 本に絞る**（REST で十分なら MQTT クライアントは載せない）。

## 検討したアプローチ

### A. スタックちゃんファームが HA REST API で取得（推奨候補）

独り言タイマーが発火する直前（または一定間隔でキャッシュ更新）に、**`GET /api/states/sensor.kanden_fatigue`** 等で数値を取得。ローカルで **帯判定**し、`exec_chatGPT` に渡す文字列を **テンプレート合成**する。

- **Pros:** HA が既にセンサーを持つだけでよい。追加サーバ不要。デバッグはブラウザや `curl` と同じ API。
- **Cons:** **Long-Lived Access Token** をデバイスに保持する必要（NVS や Web 設定の拡張）。トークン漏えい時の影響大。HA へ到達不能時のフォールバック設計が必要。
- **向いているとき:** 同一 LAN、HA の REST が安定しており、ファーム改修を許容できる場合。

### B. HA が「次の話題ヒント」をスタックちゃんへプッシュ

HA の自動化やスクリプトで、`sensor.kanden_fatigue` 更新時（または周期）に **`GET /chat` や新設 `GET /monologue_hint?...`** で **短文プロンプトだけ**をデバイスに渡し、デバイスは **キューまたは NVS** に保持。独り言発火時はそのヒントを優先。

- **Pros:** スタックちゃんから **HA 認証情報を持たせなくてよい**（プッシュは匿名 HTTP のままにしやすい）。ロジックの大半を **YAML で変更可能**。
- **Cons:** 「独り言のタイミング」と「HA 更新」の **同期ズレ**。新エンドポイントや状態保持が必要。HA からデバイスがオフラインのときの扱い。

### C. 同一 LAN の小さなプロキシ（rp5 上の HTTP）

スタックちゃんは **`http://rp5.local:9xxx/fatigue`** のみ叩く。rp5 上のプロセスが **HA に問い合わせ**し、**数値または帯ラベルだけ**を返す。

- **Pros:** トークンを **RPi 側に閉じる**。ファームはシンプルな GET のみ。
- **Cons:** 運用コンポーネントが増える。プロキシの可用性・セキュリティ（LAN 内制限）が課題。

**採用:** ユーザー回答により **アプローチ A** を第一候補として確定（下記 Resolved Questions 参照）。

## Key Decisions

| 項目 | 決定内容 |
|------|----------|
| データ経路 | **A:** スタックちゃんファームが **Home Assistant REST API** で `sensor.kanden_fatigue` を取得する |
| 認証 | デバイスに **Long-Lived Access Token** を保持する（**許容: はい**）。保存先・入力 UI は計画で定義（NVS、`/apikey` 拡張など） |
| 話題トーン | **高疲労:** 寄り添い・共感中心。**低疲労:** 明るい冗談・軽い雑談。**しきい値:** `>= 0.7` / `< 0.7` の **2 帯**（計画書で確定） |
| 更新頻度 | **独り言が発火するたびに** HA へ問い合わせる（キャッシュなし。タイムアウト時の挙動は計画で定義） |
| OpenAI への載せ方 | **計画で確定:** 数値は送らず、**口調用の日本語プレフィックスのみ**（詳細は計画書） |
| フォールバック | **計画で確定:** 失敗時は **`random_words` のみ** |

## Resolved Questions

**ユーザー回答（2026-03-23）**

1. **データ取得の主経路:** **A**（デバイスが HA REST で取得）。
2. **認証方針:** デバイスに **HA 長期トークンを載せることは許容（はい）**。
3. **帯と文言:** **高疲労は寄り添い系、低疲労は明るい冗談・軽い話題。**
4. **更新頻度:** **独り言のたびに取得**（毎回問い合わせ）。

**計画フェーズで確定（2026-03-23）**

5. **しきい値:** **2 帯**、`>= 0.7` を高疲労（寄り添い）、`< 0.7` を低疲労（明るい冗談）。既存 HA しきい値と整合。
6. **OpenAI 入力:** **疲労数値はモデルに送らない**。プレフィックスは定性のみ。
7. **HA 取得失敗時:** **`random_words` のみ**で `exec_chatGPT`。
